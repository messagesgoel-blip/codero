# Contract: Structured Per-Step Gate-Check Log / Heartbeat Schema (LOG-001)

**Status:** implemented — pending merge (COD-057)
**Sprint:** COD-057
**Depends on:** GC-001 (`docs/contracts/gate-check-schema-v1.md`)
**Packages:** `internal/gatecheck`, `internal/tui/adapters`

---

## Purpose

Define a normalized, deterministic **display-state** model for every step in the
pre-commit gate pipeline so that the dashboard Findings tab and TUI can render
each step as `passing`, `failing`, or `disabled` **without inferring state from
freeform text or from raw engine status strings**.

This contract does **not** replace or modify GC-001. GC-001 remains the single
authoritative wire format for the JSON report envelope. LOG-001 defines only the
normalized presentation layer on top of it.

---

## Canonical Display States

| `display_state` | Meaning |
|---|---|
| `passing` | The check ran and produced a passing result. No action needed. |
| `failing` | The check ran and found a problem. Action required. |
| `disabled` | The check was not run (disabled or skipped for any reason). |

### Mapping from GC-001 `status`

| Raw `status` (GC-001) | `display_state` (LOG-001) |
|---|---|
| `pass` | `passing` |
| `fail` | `failing` |
| `skip` | `disabled` |
| `disabled` | `disabled` |
| *(any unknown value)* | `disabled` |

**Rationale for `skip` → `disabled`:** from the UI perspective a step that did
not run is not actionable regardless of *why* it was skipped. The raw `status`
(`skip` vs `disabled`) and the `reason_code`/`reason` fields carry the
distinction for drill-down views that need it.

## Stable Step Identifiers

The canonical pre-commit step matrix is fixed in runner order. The IDs are
**stable** — consumers must not hard-code the *count* of steps because optional
checks may be added or removed between releases.

| Order | `id` | Name | Group | Required |
|---|---|---|---|---|
| 1 | `file-size` | File size limit | format | yes |
| 2 | `merge-markers` | Merge conflict markers | format | yes |
| 3 | `trailing-whitespace` | Trailing whitespace | format | no |
| 4 | `final-newline` | Missing final newline | format | no |
| 5 | `forbidden-paths` | Forbidden path blocker | config | yes |
| 6 | `config-validation` | Config file validation (JSON/YAML) | config | yes |
| 7 | `lockfile-sync` | Lockfile sync check | config | no |
| 8 | `exec-bit-policy` | Executable bit policy | config | no |
| 9 | `gofmt` | Go formatting (gofmt) | format | no |
| 10 | `gitleaks-staged` | Secret scan (gitleaks) | security | yes |

The authoritative ordering is enforced in `internal/gatecheck/engine.go`
`buildRunners()`. Tests in `engine_test.go` (`TestEngine_StableStepOrder`)
assert this ordering does not regress.

## Reason Preservation

For `disabled` display-state steps the `reason_code` and `reason` fields from
the GC-001 `CheckResult` **must** be preserved verbatim. Consumers must not
substitute or synthesize reason text.

Rules (from the engine normalization pass, inherited by LOG-001 adapters):

1. If `status` is `skip` or `disabled` and `reason_code` is empty, the engine
   sets `reason_code = "not_applicable"`.
2. If `reason` is empty but `reason_code` is non-empty, the display adapter may
   expand `reason_code` as the reason string (same as GC-001's `DisplayReason`).
3. The full GC-001 reason code vocabulary applies:
   `user_disabled`, `missing_tool`, `not_applicable`, `not_in_scope`,
   `timeout`, `infra_bypass`, `infra_auth`, `infra_rate_limit`,
   `infra_network`, `exec_error`, `check_failed`.

## Implementation

### Go — `internal/gatecheck` (`model.go`)

```go
type DisplayState string

const (
    DisplayPassing  DisplayState = "passing"
    DisplayFailing  DisplayState = "failing"
    DisplayDisabled DisplayState = "disabled"
)

func (s CheckStatus) ToDisplayState() DisplayState {
    switch s {
    case StatusPass:    return DisplayPassing
    case StatusFail:    return DisplayFailing
    default:            return DisplayDisabled // skip, disabled, unknown
    }
}
```

### Go — TUI adapter (`internal/tui/adapters/gate.go`)

`CheckViewModel.DisplayState` (`string`) is populated by `FromCheckReport()`:

```go
DisplayState: string(c.Status.ToDisplayState()),
```

### Go — TUI snapshot renderer (`internal/tui/render_gatecheck.go`)

The snapshot table now includes a `DISPLAY` column between `GROUP` and `STATUS`,
showing the normalized display state for each step.

### JavaScript — dashboard (`internal/dashboard/static/index.html`)

```javascript
// LOG-001: maps raw GC-001 status → normalized display state.
function toDisplayState(status) {
  if (status === 'pass') return 'passing';
  if (status === 'fail') return 'failing';
  return 'disabled';
}
```

The `renderFindings()` function uses `toDisplayState()` to derive the CSS class
for the RESULT column. If the check object carries a `display_state` field
(future server-side normalization), that value takes precedence.

---

## Non-Changes (Backward Compatibility)

- The GC-001 JSON wire format is **unchanged**. No new fields are added to the
  report envelope or `CheckResult` struct.
- The `/api/v1/dashboard/gate-checks` endpoint continues to pass through the
  raw GC-001 report as a `json.RawMessage`.
- `overall_status` semantics (`pass`/`fail`) are unchanged.
- Exit codes 0, 1, and 2 are unchanged.
