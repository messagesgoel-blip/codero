## Scope Summary

v1.2.4 focuses on the TUI and dashboard visual refresh work explicitly deferred from v1.2.3,
plus three feedback-loop hardening items surfaced by the TEST-1/TEST-2/full-cycle pilots:
- COD-NEW-A: post-push CI watcher
- COD-NEW-B: pre-push test gate
- COD-NEW-C: autonomous finish-loop orchestrator (codero-finish.sh)

v1.2.3 delivered functional correctness (BUG-001 fix, surface parity harness). v1.2.4 delivers
the operator-facing visual quality improvements that were out of scope for v1.2.3, closes
the CI feedback gap with finish-loop automation, and defines the structured heartbeat/log
shape needed for the UI to render per-step passing/failing/disabled states and current
activity/session context.
The per-step feed should come from the canonical gate-check report (`/api/v1/dashboard/gate-checks`),
not a parallel ad hoc `gate-steps` surface. Current activity should come from fresh
branch-heartbeat rows, with task/phase context filled in only when the backend can resolve it.

### COD-NEW-A: Post-Push CI Watcher (P1)
**Surface:** Agent toolkit — `ci-watch.sh`
**Owner:** ClaudeCode
**Status:** Completed (merged PR #60)
**Surfaced by:** TEST-2 local gate pilot (2026-03-18) — CI failure post-push had no feedback path back to agent

**Description:** After `git push`, Codero has no mechanism to monitor GitHub Actions status and
surface CI failures back to the agent in-session. The agent only learns of CI failures by
manually polling `gh run list`. This item adds a `ci-watch.sh` script to the shared
agent-toolkit that polls CI status until completion and delivers a structured pass/fail report
with failure log excerpts.

**Acceptance criteria:**
- `ci-watch.sh [branch] [timeout_seconds]` polls `gh run list` every 15s until all runs complete or timeout
- On success: prints summary table of all checks with durations
- On failure: prints failing job names + first 40 lines of failure log for each failing job
- Exit code 0 for all-pass, 1 for any failure, 2 for timeout
- Installed at `/srv/storage/shared/agent-toolkit/bin/ci-watch.sh`
- Integrated into `agent-finish-task` workflow (optional, post-push poll)

### COD-NEW-B: Pre-Push Test Gate (P1)
**Surface:** Git hook — `pre-push`
**Owner:** ClaudeCode
**Status:** Completed (merged PR #60)
**Surfaced by:** TEST-2 local gate pilot (2026-03-18) — `TestIntentionalFail` passed pre-commit
(which does not run `go test`) and reached CI undetected

**Description:** The pre-commit hook runs gitleaks, semgrep, and the AI review gate, but does NOT
run `go test ./...`. A deliberately broken test therefore sails through pre-commit and only fails
in GitHub Actions CI 30–60 seconds later. This item adds a `pre-push` hook that runs the full
Go test suite before any push is allowed to reach the remote, closing the gap between
pre-commit and CI.

**Acceptance criteria:**
- `/srv/storage/shared/agent-toolkit/hooks/pre-push` runs `go test ./...` for Go repos (detected via `go.mod` at repo root)
- Push is blocked (exit 1) if any test fails; full test output shown
- Push proceeds (exit 0) if all tests pass
- Hook is a no-op for non-Go repos
- Symlinked from `.githooks/pre-push` via `install-hooks`
- `install-hooks` script updated to install `pre-push` alongside `pre-commit`

### COD-NEW-C: Autonomous Finish-Loop Orchestrator (P0)
**Surface:** Agent toolkit — `codero-finish.sh`
**Owner:** ClaudeCode
**Status:** Completed (merged PR #62)
**Surfaced by:** Full-cycle pilot (2026-03-18) — agent still manually drove each step
(commit, push, PR open, ci-watch call, CodeRabbit poll, feedback read, re-commit).
The intent is for Codero to own all of that once the agent says "I'm done."

**Description:** `codero-finish.sh "<commit-message>"` is the single handoff point.
The agent writes code, stages changes, then calls this script. Codero takes over:

1. **Commit** — runs `git commit` triggering pre-commit gates in repo-default order
   (as defined by the orchestrator, e.g. `two-pass-review.sh`): gitleaks, path-guard,
   Semgrep (always a mandatory blocking gate), AI gate. If blocked, returns exact gate
   output as structured feedback and exits 3.
2. **Push** — runs `git push` triggering pre-push (`go test ./...`).
   If blocked, returns test failure output and exits 4.
3. **PR** — creates PR if none exists (with `@coderabbitai review` in body).
   If PR exists, posts `@coderabbitai review` comment to trigger fresh review.
4. **CI watch** — polls `gh run list` until all runs complete (via `ci-watch.sh`).
   Hard-fails only on non-infrastructure failures (sonarcloud infra failures ignored).
5. **CodeRabbit poll** — waits up to 10 minutes for CodeRabbit to complete its review.
6. **Feedback extraction** — parses actionable inline comments from the PR review thread.
7. **Deliver** — writes `.codero/finish-feedback.md` + prints to stdout.
   Exit 0 = merge-ready (all clean). Exit 1 = feedback for agent to act on.

Agent addresses feedback, re-stages, calls `codero-finish.sh` again. Loop repeats
until exit 0 (all gates pass, CI clean, no actionable CodeRabbit comments).

**Acceptance criteria:**
- Installed at `/srv/storage/shared/agent-toolkit/bin/codero-finish.sh`
- Exit 0: all gates pass + CI clean + 0 actionable CodeRabbit comments
- Exit 1: structured feedback in `.codero/finish-feedback.md` and stdout
- Exit 2: CI non-infrastructure failure with log excerpt
- Exit 3: pre-commit blocked with exact gate output
- Exit 4: pre-push blocked with test failure output
- Feedback file is cleared on clean exit
- Does not require agent to manually call `ci-watch.sh`, open PR, or poll CodeRabbit

### COD-055: Finish-Loop Stabilization Follow-Up (P0)
**Surface:** Agent toolkit — `codero-finish.sh`
**Owner:** ClaudeCode
**Status:** In progress (PR #63 open)
**Surfaced by:** CODERO-E2E-CTX-001 — Phase 5 timed out because CodeRabbit completion was only
checked through `reviews[]`, while CodeRabbit sometimes finished via PR issue comments.
The same run also staged `SCRATCHPAD.md` because auto-stage relied on `git add -u` without a
pre-commit purge.

**Description:** Hardened the finish loop so CodeRabbit completion is detected from both the
review thread and the PR comment stream, and so scratchpad files are removed from the staged
set before commit every time.

**Acceptance criteria:**
- CodeRabbit terminal completion is detected from both review records and PR comments
- `SCRATCHPAD.md` cannot be auto-staged or committed by the finish loop
- Merge-ready runs return exit `0` when CI is green and CodeRabbit reports no actionable comments
- The finish loop still produces actionable feedback when CodeRabbit does have findings
