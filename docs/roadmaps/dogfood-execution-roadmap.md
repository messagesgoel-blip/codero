# Codero Execution Roadmap — Dogfood-First, OpenClaw-Expanded

Status: **active**
Owner: sanjay
Updated: 2026-04-03
Supersedes: `agent-task-execution-roadmap.md` (SUB-001 through FIN-001)
Next task: OCL-022

## Context

Codero is running (healthy, 2 containers, Redis up) but isn't in the critical path of any
real workflow. Sessions register with empty repo/branch, zero gate runs are recorded, the
scorecard is blank, and all actual work (commit, PR, review, merge) happens through normal
git/GitHub flows bypassing Codero entirely.

This roadmap restructures the execution plan to:
1. **Dogfood on the Codero repo itself** — every step is tested by using Codero to develop Codero
2. **Increase OpenClaw's role** as the advisory/delivery layer without giving it executive powers
3. **Show visible progress at every step** — each task produces observable dashboard changes
4. **Find real problems** by running the system under its own development workflow

### Actor Authority (Unchanged)

| Actor | Can | Cannot |
|---|---|---|
| **OpenClaw** | Observe agent output, package submit payloads, deliver findings to PTY, answer operator queries, log audit events | Decide merge readiness, execute merges, write to durable state, call GitHub API directly |
| **Codero** | Own state machine, evaluate merge predicates, create PRs, execute merges, run gate pipeline | Inject text into PTY directly (delegates to OpenClaw), reason about code quality |
| **Agent** | Edit files, run tools, signal TASK_COMPLETE | Commit, push, create PRs, merge |

### Dogfood Target

All tasks tested on: `codero` repo (`/srv/storage/repo/codero/.worktrees/main`)
Secondary: `whimsy` repo (`/srv/storage/repo/whimsy`)

### OpenClaw Runtime

OpenClaw is not a standalone binary on this system. The runtime components are:
- PTY bridge: `/srv/storage/shared/tools/bin/agent-tmux-bridge` (proven, 5 families)
- LiteLLM proxy: `http://localhost:4000` (model routing)
- Shadow config: `~/.openclaw-codero-smoke/openclaw.json` (version 2026.3.28)
- Codero adapter: `acpx` plugin (session lifecycle calls)

"OpenClaw" in this roadmap refers to the adapter service that ties these together.

### Decisions

- **OpenClaw adapter location:** In-repo at `services/openclaw-adapter/` (Go). Deployed as a
  Docker sidecar via `docker-compose.yml`. Same language as Codero, same CI, same PR workflow.
- **Testing:** Automated E2E tests from the start. Each task includes a Go test file under
  `tests/e2e/` or `tests/integration/` that exercises the full path. Tests run against the
  live dev Codero instance at `127.0.0.1:8110` or a test-scoped instance.

## Wave A — Codero Sees Its Own Development

**Goal:** Dashboard shows real data from Codero's own development. No spec changes, no new
subsystems. Pure wiring.

### WIRE-001: Bind sessions to repo and branch

**What:** Session registration and heartbeat hooks pass the active repo/branch to Codero so
sessions show real context instead of `repo: ""`.

**Process:**
1. Modify the Claude Code heartbeat hook (generated by `codero agent hooks --install`) to
   detect `$(git rev-parse --show-toplevel)` and `$(git branch --show-current)` and pass them
   as `--repo` and `--branch` flags to `codero session heartbeat`.
2. Modify `codero session register` to accept optional `--repo` and `--branch` flags and
   persist them to the session record.
3. If the session already has repo/branch from a prior heartbeat and the values change (agent
   switched worktrees), update the session record.

**Files:**
- `cmd/codero/session_heartbeat.go` — add `--repo`, `--branch` flags
- `cmd/codero/session_register.go` — add `--repo`, `--branch` flags
- `cmd/codero/agent_hooks.go` — update generated hook script to detect and pass repo/branch
- `internal/session/` or `internal/daemon/grpc/` — persist repo/branch on session record
- Migration: add `repo` and `branch` columns to `agent_sessions` if not already present

**Definition of Done:**
- `codero session heartbeat --repo=codero --branch=main` succeeds
- Dashboard active sessions page shows `repo: codero`, `branch: main` for the current session
- Heartbeat hook output (via `codero agent hooks --print`) includes repo/branch detection
- Unit test: session register with repo/branch persists correctly
- Unit test: heartbeat updates repo/branch on change

**E2E Test** (`tests/e2e/wire001_session_binding_test.go`):
```text
1. Call `codero session register --agent-id=e2e-test --repo=codero --branch=main`
2. Assert: API `GET /api/v1/dashboard/active-sessions` returns session with repo=codero, branch=main
3. Call `codero session heartbeat --session-id=<id> --repo=whimsy --branch=feat/test`
4. Assert: session now shows repo=whimsy, branch=feat/test (updated on change)
5. Call `codero session end --session-id=<id>`
6. Assert: session no longer in active sessions
```

### WIRE-002: Record pre-commit gate results in Codero

**What:** The existing pre-commit hook reports its pass/fail result to Codero after running,
so the dashboard gate health and scorecard reflect real data.

**Process:**
1. Add a `codero record-precommit` CLI command that accepts `--repo`, `--branch`, `--result`
   (pass/fail), `--duration-ms`, and `--checks` (comma-separated list of checks that ran).
2. This command writes a row to `review_runs` (or a new `precommit_runs` table) in the DB.
3. Append a call to `codero record-precommit` at the end of the shared pre-commit hook at
   `/srv/storage/shared/agent-toolkit/hooks/pre-commit`.
4. The call is best-effort (failures don't block the commit).

**Files:**
- `cmd/codero/record_precommit.go` — new CLI command (or extend existing if `record-precommit` exists)
- Shared hook: `/srv/storage/shared/agent-toolkit/hooks/pre-commit` — append reporting call
- `internal/state/` — insert function for precommit results
- `internal/dashboard/queries.go` — update `queryGateHealth` to include precommit data

**Definition of Done:**
- After a successful commit in the Codero repo, `codero record-precommit` is called
- Dashboard Gate Health table shows a row for "precommit" with real pass/fail counts
- Scorecard shows non-zero `pre-commit runs` count
- The commit itself is not blocked or slowed by reporting failures
- Unit test: record-precommit writes to DB correctly

**E2E Test** (`tests/e2e/wire002_precommit_recording_test.go`):
```text

4. Call `codero record-precommit --repo=codero --branch=main --result=fail --duration-ms=800 --checks=gitleaks`
```

### OCL-011: Build OpenClaw query adapter

**What:** Build a thin OpenClaw adapter service that receives natural language queries from
the dashboard chat panel, queries Codero's state endpoint (OCL-010), and returns answers
via LiteLLM.

**Process:**
1. Create a standalone service (Go or Python) under `services/openclaw-adapter/` in the Codero
   repo. It runs as a sidecar alongside the Codero daemon.
2. The adapter exposes `POST /query` accepting `{prompt, conversation_id}`.
3. On each query it:
   a. Fetches `GET /api/v1/openclaw/state` from Codero
   b. Builds a system prompt with the state context
   c. Calls LiteLLM at `http://localhost:4000` with the user prompt + context
   d. Returns the LLM response
4. The dashboard chat panel is rewired to call the adapter instead of Codero's built-in chat.

**Files:**
- `services/openclaw-adapter/main.go` (or `main.py`) — new service
- `services/openclaw-adapter/Dockerfile` — containerized deployment
- `docker-compose.yml` updates for sidecar
- `internal/dashboard/static/js/renderers/chat.js` — rewire chat endpoint

**Definition of Done:**
- Dashboard chat panel sends queries to the OpenClaw adapter
- Adapter queries Codero state API and returns LLM-powered answers
- `/status`, `/queue`, `/blocked`, `/health` quick queries work
- Fallback: if adapter is unreachable, dashboard shows "OpenClaw unavailable" message
- Adapter logs every query and response (audit trail)
- Integration test: send query, verify response includes real session data

**E2E Test** (`tests/e2e/ocl011_adapter_query_test.go`):

### OCL-021: OpenClaw adapter delivers findings to agent PTY

**What:** The OpenClaw adapter can deliver structured findings into a running agent's PTY
session using the proven `agent-tmux-bridge`.

**Process:**
1. Add `POST /api/v1/openclaw/deliver` endpoint on the adapter accepting:
   `{session_id, findings[], source}`.
2. The adapter:
   a. Looks up the session's tmux name and agent kind from Codero's active sessions API
   b. Formats findings into a readable message (file:line: message, grouped by severity)
   c. Calls `agent-tmux-bridge deliver --session <tmux> --profile <kind> --message-stdin`
   d. Logs the delivery in the audit log
3. This is a **pull model**: the adapter calls Codero for findings, then delivers. Codero
   never calls the bridge directly.

**Files:**
- `services/openclaw-adapter/deliver.go` — delivery handler
- `services/openclaw-adapter/bridge.go` — PTY bridge integration
- `services/openclaw-adapter/audit.go` — log delivery events

**Definition of Done:**
- `POST /api/v1/openclaw/deliver` with a session_id and findings delivers to the agent PTY
- Agent sees the formatted findings message in their terminal
- Delivery is logged in the audit log with: timestamp, session_id, finding count, delivery result
- If the bridge fails (session not found, agent busy), error is logged and returned
- Integration test: deliver to a test tmux session, capture output, verify message content

### OCL-022: Wire webhook findings to OpenClaw delivery

**What:** When a GitHub webhook arrives with review findings (CodeRabbit posts), Codero
normalizes them and signals OpenClaw to deliver to the agent.

**Process:**
1. In the webhook reconciler, after fetching PR state and finding unresolved review comments:
   a. Normalize findings per RF-001
   b. Store findings in `findings` table (already exists)
   c. If an active session exists for this repo/branch, call the OpenClaw adapter's
      `POST /api/v1/openclaw/deliver` with the session_id and findings
2. This is **Codero initiating delivery through OpenClaw** — Codero decides what to deliver,
   OpenClaw decides how and when to inject.

**Files:**
- `internal/webhook/reconciler.go` — add OpenClaw delivery call after finding normalization
- `internal/webhook/openclaw_client.go` — HTTP client for OpenClaw adapter
- Config: `CODERO_OPENCLAW_ADAPTER_URL` env var

**Definition of Done:**
- CodeRabbit posts a review on a tracked PR
- Codero receives the webhook, normalizes findings, stores them
- Codero calls OpenClaw adapter to deliver findings to the agent session
- Agent sees the findings in their PTY
- Dashboard activity feed shows the delivery event
- If no active session exists for the branch, findings are stored but not delivered (no error)
- If OpenClaw adapter is unreachable, findings are stored, delivery is skipped with a warning log
- Integration test: mock webhook → findings stored → delivery triggered

## Wave D — Submission Pipeline (Simplified)

**Goal:** Agents submit work through Codero instead of manually calling `gh pr create`.
Codero commits, pushes, creates the PR. OpenClaw observes TASK_COMPLETE and triggers submit.

### SUB-010: Add codero submit command

**What:** A `codero submit` CLI command that commits staged changes, pushes the branch, and
creates or updates a PR. This is the simplified version of SUB-005 — no Submission Record
yet, just the mechanical flow.

**Process:**
1. Add `codero submit --repo=<repo> --branch=<branch> --title=<title> --body=<body>` command.
2. The command:
   a. Verifies staged changes exist (rejects if clean worktree)
   b. Commits with authorship policy (signed, agent as author)
   c. Pushes to origin
   d. Creates PR via GitHub API (or updates existing)
   e. Requests reviewers (from config, not CODEOWNERS yet)
   f. Records the PR in branch_states (like WIRE-003 but automatic)
   g. Updates the session's assignment state to `submitted`
3. This replaces the manual `git commit && git push && gh pr create` flow.

**Files:**
- `cmd/codero/submit_cmd.go` — extend existing (it exists but may be incomplete)
- `internal/gitops/` — commit and push functions (already exist)
- `internal/github/client.go` — PR creation (already exists: `CreatePR`, `FindPR`)
- `internal/state/` — update assignment state on submit

**Definition of Done:**
- `codero submit` from a Codero worktree with staged changes creates a PR
- PR title and body come from CLI flags
- PR appears in dashboard Pipeline view
- Reviewers are requested (from config list)
- If PR already exists for the branch, it is found and reused (not duplicated)
- If worktree is clean, command returns error "no changes to submit"
- Unit test: submit with mocked GitHub client creates PR
- Integration test: submit updates assignment state

### SUB-011: Add Submission Record (lightweight)

**What:** Track each submit attempt as a row in a new `submissions` table. This is the
simplified version of SUB-002 — fewer fields than the spec, enough for dedup and lineage.

**Process:**
1. Add migration creating `submissions` table with:
   `submission_id, assignment_id, session_id, repo, branch, head_sha, diff_hash,
    attempt_local (default 0), attempt_remote (default 0), state, result, created_at, updated_at`
2. `codero submit` creates a Submission Record before committing.
3. Duplicate detection: reject if same `(diff_hash, head_sha)` already exists for this assignment.
4. The record is updated as the submission progresses through the pipeline.

**Files:**
- `internal/state/migrations/000025_submissions.up.sql` — new table
- `internal/state/submissions.go` — CRUD functions
- `cmd/codero/submit_cmd.go` — create submission record on submit

**Definition of Done:**
- Each `codero submit` creates a Submission Record with unique submission_id
- `diff_hash` is computed from `git diff --staged` SHA
- Duplicate submit (same diff, same HEAD) is rejected with clear error
- Dashboard Pipeline view shows submission_id and attempt counts

### SUB-012: OpenClaw observes TASK_COMPLETE and triggers submit

**What:** When an agent outputs `TASK_COMPLETE` in its PTY, OpenClaw observes it, extracts

3. It calls `codero submit` with the extracted title and body.

**Files:**

**Definition of Done:**
- Agent outputs `TASK_COMPLETE` with summary block → OpenClaw detects it within 10 seconds
- OpenClaw extracts pr_title, change_summary, test_notes
- OpenClaw calls `codero submit` with extracted metadata
- If no summary block, fallback title/body is used
- If `codero submit` rejects (duplicate, clean worktree), OpenClaw logs the rejection
- Audit log records: TASK_COMPLETE detected, summary extracted, submit triggered, result
- Unit test: parser extracts summary block correctly
