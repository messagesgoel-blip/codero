# v1.2.1 Post-Release Stabilization Watch

## Purpose

This runbook governs the 48-hour operational watch following v1.2.1 promotion.
Its goal is to detect regressions against the v1.2.0 baseline before operators
fully commit to the new release, and to provide explicit backout triggers if
any regression is found.

---

## Monitoring Window

| Property | Value |
|---|---|
| Start | v1.2.1 tag push (2026-03-17) |
| Duration | 48 hours (ends 2026-03-19 ~12:00 UTC) |
| Cadence | Every 6 hours (CI: `stabilization-watch.yml`) |
| Owner | Codero maintainer with `main` write access |
| Escalation | If any FAIL appears, treat as incident until root-cause confirmed |

---

## Manual Execution

```bash
cd /srv/storage/repo/codero

# Gather v1.2.1 status
CODERO_BIN=/srv/storage/shared/tools/releases/codero-v1.2.1/codero \
  bash scripts/automation/release-status.sh \
    --version v1.2.1 --json \
    2>v121-status.txt | tee v121-status.json

# Gather v1.2.0 baseline
CODERO_BIN=/srv/storage/shared/tools/releases/codero-v1.2.0/codero \
  bash scripts/automation/release-status.sh \
    --version v1.2.0 --json \
    2>v120-status.txt | tee v120-status.json

# Triage (COD-045): classify checks, emit counts, exit 1 on NEW_FAIL
bash scripts/automation/stabilization-triage.sh \
  --current  v121-status.json \
  --baseline v120-status.json \
  --output   both \
  2>triage-human.txt | tee triage.json
triage_exit=${PIPESTATUS[0]}
echo "triage exit: ${triage_exit}"

# Detailed diff (supplementary)
bash scripts/automation/compare-release-status.sh \
  --new      v121-status.json \
  --baseline v120-status.json \
  --output   both

# Exit code 0 = regression-free; exit code 1 = new FAILs detected
echo "compare exit: $?"
```

### Triage with previous run (persistence detection)

When a second watch run is available, pass it via `--previous` to mark failures
that were already present in the prior run as "persistent":

```bash
bash scripts/automation/stabilization-triage.sh \
  --current  v121-status.json \
  --baseline v120-status.json \
  --previous v121-status-prev.json \
  --output   both
```

A `persistent: true` annotation in the triage output means the failure also
appeared in the previous watch run — a strong signal that the issue is not
transient and backout criteria apply.

## Triage Automation (COD-045)

The `stabilization-triage.sh` script is the primary regression signal for each
watch run. It compares the current release-status JSON against the v1.2.0 baseline
and classifies every check:

| Category | Meaning | Workflow result |
|---|---|---|
| `NEW_FAIL` | Check is FAIL in v1.2.1, was PASS/SKIP in v1.2.0 | **Workflow fails; issue escalation triggered** |
| `MISSING_FROM_CURRENT` | Check exists in baseline but is missing from current report | **Workflow fails; issue escalation triggered** |
| `NEW_SKIP` | Check is SKIP in v1.2.1, was PASS in v1.2.0 (env drift) | Warning annotation; workflow passes |
| `RECOVERED` | Check is PASS/SKIP in v1.2.1, was FAIL in v1.2.0 | Info; logged as positive signal |
| `UNCHANGED` | Status same in current and baseline | None |

**`persistent: true`** in a `NEW_FAIL` entry means the same check also failed
in the previous watch run — a strong indicator the failure is not transient.

Triage output files per run (in CI artifacts):
- `triage.json` — machine-readable; key fields: `regression_free`, `new_fails[]`, `totals`
- `triage-human.txt` — human-readable markdown table

## Automated Issue Lifecycle (COD-046 / COD-047)

The workflow manages a single GitHub Issue per version that tracks the full
regression/recovery lifecycle automatically.

### Issue lifecycle

| Event | Workflow action |
|---|---|
| First `regression_free=false` run | Creates `v1.2.1 Stabilization Regression` issue with failing check IDs, run URL, first-seen timestamp, runbook link, and a hidden **regression signature** |
| Subsequent run with **same** regression set | Anti-noise guard: skips duplicate comment (signature unchanged) |
| Subsequent run with **different** regression set | Adds update comment with new failing check IDs and updated signature |
| First `regression_free=true` run after regression | Posts recovery summary comment (counts + run URL + timestamp) and **closes the issue automatically** |

### Regression signature

Each escalation comment embeds a deterministic regression signature as an HTML
comment in the body:

```html
<!-- regression-sig: C-001,C-002,V-003 -->
```

The signature is the sorted, comma-joined union of all `NEW_FAIL` and
`MISSING_FROM_CURRENT` check IDs. The anti-noise guard extracts this marker
from the most recent escalation (issue body or last comment) and compares it
to the current run's signature. If they match, no new comment is posted.

### Manual override

If you need to close the issue before recovery is confirmed by CI:

```bash
gh issue close <NUMBER> --repo <ORG/REPO> --comment "Manual close: <reason>"
```

If you need to re-open a closed issue after a new regression:

```bash
gh issue reopen <NUMBER> --repo <ORG/REPO>
```

The automated workflow will continue to create new issues (or comments) on
future regression runs regardless of manual close/reopen state — the issue
lookup checks for **open** issues only.

### Token and permissions

Uses the built-in `GITHUB_TOKEN` with `issues: write` permission —
no hardcoded secrets.

## Pass / Fail / Backout Criteria

### Stabilization PASS (continue)

All of the following must hold:

| Condition | Check |
|---|---|
| v1.2.1 `overall_status=PASS` | Zero FAILs in `stabilization-v121.json` |
| `regression_free=true` in `triage.json` | No NEW_FAIL or MISSING_FROM_CURRENT entries |
| NEW_SKIP entries have documented reason | e.g. daemon checks SKIP in CI — expected |
| Workflow green for 3+ consecutive 6h runs | Check GitHub Actions history |

### Stabilization FAIL (investigate)

Any of the following triggers an investigation:

| Trigger | Severity | Action |
|---|---|---|
| Any check in `prove` group → `NEW_FAIL` | Critical | Immediate backout consideration |
| Any check in `verify` group → `NEW_FAIL` (non-daemon) | Critical | Immediate backout consideration |
| Any baseline check `MISSING_FROM_CURRENT` | Critical | Immediate backout consideration |
| `drift.checksum_match=false` | Critical | Binary integrity breach — backout immediately |
| `triage.json` shows `regression_free=false` | High | Triage within 1h; check escalation issue |
| Same `NEW_FAIL` in two consecutive runs (`persistent: true`) | High | Execute rollback |
| `NEW_SKIP` count increases unexpectedly | Medium | Document and monitor |

### Backout Triggers (execute rollback)

Immediately execute rollback if **any** of these are true:

1. Any `prove` check (`C-001..C-022`) FAILs that passed in v1.2.0.
2. Any `verify` check (`V-001..V-006`) FAILs — these are non-daemon, always expected to pass.
3. `drift.checksum_match=false` — artifact tamper or build corruption.
4. Two consecutive 6h watch runs show the same `NEW_FAIL` or `MISSING_FROM_CURRENT` regression (not a transient).

### Expected SKIPs (not a backout trigger)

| Check | Reason |
|---|---|
| C-004 `race-tests` | `--fast` flag set in CI |
| C-007..C-009 `obs-*` | Daemon not running in CI |
| V-007..V-009 `endpoint-*` | Daemon absent in CI |
| DR-V02 `drift-artifact-sha` | `/srv/storage/` not mounted in CI runner |

## Comparison Interpretation Guide

### Triage output (`triage.json`) — primary signal

| Category | Meaning | Action |
|---|---|---|
| `NEW_FAIL` | PASS/SKIP → FAIL: regression | **Investigate immediately; consider backout** |
| `MISSING_FROM_CURRENT` | Baseline check not present in current report | **Investigate immediately; consider backout** |
| `NEW_SKIP` | PASS → SKIP: env drift | Document reason; not a failure |
| `RECOVERED` | FAIL → PASS/SKIP: improvement | Log as positive signal |
| `UNCHANGED` | No change (includes FAIL→FAIL, PASS→PASS, etc.) | None |

### Detailed diff (`stabilization-diff.json`) — supplementary

The `compare-release-status.sh` script produces a finer-grained delta for audit
purposes. Its categories overlap with triage but add `DEGRADED`, `IMPROVED`, and
`NEW_CHECK` distinctions. Use this for root-cause analysis, not for pass/fail decisions.

| Delta Type | Meaning | Action |
|---|---|---|
| `STABLE` | Status unchanged between releases | None |
| `NEW_FAIL` | PASS/SKIP → FAIL: regression | **Investigate immediately; consider backout** |
| `RECOVERED` | FAIL → PASS/SKIP: improvement | Log as positive signal |
| `DEGRADED` | PASS → SKIP: environmental gap widened | Document reason; not a failure |
| `IMPROVED` | SKIP → PASS: environment improvement | Log as positive signal |
| `NEW_CHECK` | Check exists in new, absent in baseline | Expected when checks are added |

**Totals delta interpretation (diff):**

- `pass delta > 0`: more checks passing — positive
- `fail delta > 0`: **regression** — `NEW_FAIL` entries will identify which checks
- `skip delta > 0`: more skips — check for `DEGRADED`/`NEW_SKIP` entries; document if unexpected
- `total delta > 0`: new checks added — expected when tooling evolves

## Rollback / Backout Commands

```bash
# Atomic rollback to v1.2.0
BINDIR=/srv/storage/shared/tools/bin

cp -p "${BINDIR}/codero.prev.bak" "${BINDIR}/codero.rollback.tmp"
chmod --reference="${BINDIR}/codero" "${BINDIR}/codero.rollback.tmp"
mv "${BINDIR}/codero.rollback.tmp" "${BINDIR}/codero"

codero version  # must print v1.2.0

# Verify rollback succeeded
CODERO_BIN="${BINDIR}/codero" \
  bash /srv/storage/repo/codero/scripts/automation/verify-release.sh \
    --version v1.2.0 --fast
```

The rollback is atomic (same-filesystem `mv`). No database changes occur during a patch
binary swap; no daemon restart is required unless the daemon was running during the swap.
