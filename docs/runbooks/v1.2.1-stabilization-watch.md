# v1.2.1 Post-Release Stabilization Watch

## Purpose

This runbook governs the 48-hour operational watch following v1.2.1 promotion.
Its goal is to detect regressions against the v1.2.0 baseline before operators
fully commit to the new release, and to provide explicit backout triggers if
any regression is found.

---

## Monitoring Window

| Property | Value |
|---|---|
| Start | v1.2.1 tag push (2026-03-17) |
| Duration | 48 hours (ends 2026-03-19 ~12:00 UTC) |
| Cadence | Every 6 hours (CI: `stabilization-watch.yml`) |
| Owner | Codero maintainer with `main` write access |
| Escalation | If any FAIL appears, treat as incident until root-cause confirmed |

---

## Cadence and Ownership

| Slot | Approx UTC | Action |
|---|---|---|
| T+0h | 2026-03-17 12:00 | Manual dispatch after promotion |
| T+6h | 2026-03-17 18:00 | Automated (schedule) |
| T+12h | 2026-03-18 00:00 | Automated |
| T+18h | 2026-03-18 06:00 | Automated |

| T+30h..T+48h | — | Automated |
| T+48h | 2026-03-19 12:00 | Final window close — disable schedule or leave as ongoing watch |

The schedule cron (`0 */6 * * *`) can be removed or changed to daily once the
48h window closes without incidents.

## Manual Execution

```bash
cd /srv/storage/repo/codero

# Gather v1.2.1 status
CODERO_BIN=/srv/storage/shared/tools/releases/codero-v1.2.1/codero \
  bash scripts/automation/release-status.sh \
    --version v1.2.1 --json \
    2>v121-status.txt | tee v121-status.json

# Gather v1.2.0 baseline
CODERO_BIN=/srv/storage/shared/tools/releases/codero-v1.2.0/codero \
  bash scripts/automation/release-status.sh \
    --version v1.2.0 --json \
    2>v120-status.txt | tee v120-status.json

# Compare
bash scripts/automation/compare-release-status.sh \
  --new      v121-status.json \
  --baseline v120-status.json \
  --output   both

# Exit code 0 = regression-free; exit code 1 = new FAILs detected
echo "compare exit: $?"
```

## Pass / Fail / Backout Criteria

### Stabilization PASS (continue)

All of the following must hold:

| Condition | Check |
|---|---|
| v1.2.1 `overall_status=PASS` | Zero FAILs in `stabilization-v121.json` |
| `regression_free=true` | No check transitioned from pass/skip → fail vs v1.2.0 |
| DEGRADED SKIPs have documented reason | e.g. daemon checks SKIP in CI — expected |
| Workflow green for 3+ consecutive 6h runs | Check GitHub Actions history |

### Stabilization FAIL (investigate)

Any of the following triggers an investigation:

| Trigger | Severity | Action |
|---|---|---|
| Any check in `prove` group goes FAIL | Critical | Immediate backout consideration |
| Any check in `verify` group goes FAIL (non-daemon) | Critical | Immediate backout consideration |
| `drift.checksum_match=false` | Critical | Binary integrity breach — backout immediately |
| `release-status.sh --version v1.2.1` exits non-zero | High | Triage within 1h |
| `compare-release-status.sh` exits non-zero (new_fails > 0) | High | Triage within 1h |
| DEGRADED count increases unexpectedly | Medium | Document and monitor |

### Backout Triggers (execute rollback)

Immediately execute rollback if **any** of these are true:

1. Any `prove` check (`C-001..C-022`) FAILs that passed in v1.2.0.
2. Any `verify` check (`V-001..V-006`) FAILs — these are non-daemon, always expected to pass.
3. `drift.checksum_match=false` — artifact tamper or build corruption.
4. Two consecutive 6h watch runs show the same new FAIL (not a transient).

### Expected SKIPs (not a backout trigger)

| Check | Reason |
|---|---|
| C-004 `race-tests` | `--fast` flag set in CI |
| C-007..C-009 `obs-*` | Daemon not running in CI |
| V-007..V-009 `endpoint-*` | Daemon absent in CI |
| DR-V02 `drift-artifact-sha` | `/srv/storage/` not mounted in CI runner |

## Comparison Interpretation Guide

The `compare-release-status.sh` script produces a delta object with these categories:

| Delta Type | Meaning | Action |
|---|---|---|
| `STABLE` | Status unchanged between releases | None |
| `NEW_FAIL` | PASS/SKIP → FAIL: regression | **Investigate immediately; consider backout** |
| `RECOVERED` | FAIL → PASS/SKIP: improvement | Log as positive signal |
| `DEGRADED` | PASS → SKIP: environmental gap widened | Document reason; not a failure |
| `IMPROVED` | SKIP → PASS: environment improvement | Log as positive signal |
| `NEW_CHECK` | Check exists in new, absent in baseline | Expected when checks are added |

**Totals delta interpretation:**

- `pass delta > 0`: more checks passing — positive
- `fail delta > 0`: **regression** — `NEW_FAIL` entries will identify which checks
- `skip delta > 0`: more skips — check for `DEGRADED` entries; document if unexpected
- `total delta > 0`: new checks added — expected when tooling evolves

## Rollback / Backout Commands

```bash
# Atomic rollback to v1.2.0
BINDIR=/srv/storage/shared/tools/bin

cp -p "${BINDIR}/codero.prev.bak" "${BINDIR}/codero.rollback.tmp"
chmod --reference="${BINDIR}/codero" "${BINDIR}/codero.rollback.tmp"
mv "${BINDIR}/codero.rollback.tmp" "${BINDIR}/codero"

codero version  # must print v1.2.0

# Verify rollback succeeded
CODERO_BIN="${BINDIR}/codero" \
  bash /srv/storage/repo/codero/scripts/automation/verify-release.sh \
    --version v1.2.0 --fast
```

The rollback is atomic (same-filesystem `mv`). No database changes occur during a patch
binary swap; no daemon restart is required unless the daemon was running during the swap.
