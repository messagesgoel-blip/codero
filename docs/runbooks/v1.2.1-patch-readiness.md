# v1.2.1 Patch Readiness Runbook

## Overview

This runbook defines the go/no-go criteria, evidence requirements, and release procedure
for the v1.2.1 patch release of Codero. It is the authoritative reference for anyone
executing or approving the patch cut.

This document does **not** execute the release. It defines the conditions under which
the release is safe to cut.

## Checks Performed

| Check ID | Name | Description | SKIP condition |
|---|---|---|---|
| DR-001 | build-binary | `go build` from current source with version ldflags | Never skipped — fatal if fails |
| DR-002 | sha256 | SHA-256 of built binary computed and recorded | Never skipped |
| DR-003 | unit-tests | `go test -count=1 ./...` | Never skipped |
| DR-004 | race-tests | `go test -race ./...` | Never skipped |
| DR-005 | e2e-tests | `go test -tags=e2e ./tests/e2e/` | SKIP if `tests/e2e/` absent or `CODERO_E2E_SKIP=true` |
| DR-006 | prove-gate | `codero prove --fast --json` | Never skipped |
| DR-007 | verify-release | `scripts/automation/verify-release.sh` | Never skipped (V-007..V-009 inside it SKIP if daemon absent) |
| DR-008 | install-drill | Copy binary to `.dryrun-test/`, version check, checksum, cleanup | Never skipped |
| DR-009 | rollback-drill | Locate and exec base-version binary | SKIP if base artifact and `codero.prev.bak` absent |

## PASS / FAIL / SKIP Semantics

| Status | Meaning | Blocks release? |
|---|---|---|
| **PASS** | Check succeeded with expected output | — |
| **FAIL** | Check produced unexpected result or non-zero exit | Yes — must be resolved before promoting |
| **SKIP** | Check could not run due to explicit environmental gap (not a failure) | No — but SKIP reason must be documented in evidence |

**Rules:**
- A dry-run with `overall_status=PASS` means zero FAILs (any SKIPs are acceptable).
- A dry-run with `overall_status=FAIL` blocks promotion until root cause is fixed and dry-run reruns clean.
- Every SKIP must have a recorded reason in the dry-run JSON (`detail` field).
- SKIPs that are **not** expected (e.g., unit tests unexpectedly skipped) must be treated as FAILs.

## Go / No-Go Matrix

Promotion to a live `v1.2.1` release requires **all** of the following:

| Gate | Condition | Status |
|---|---|---|
| G-01 | DR-001 build-binary: PASS | Required |
| G-02 | DR-002 sha256: PASS (checksum recorded) | Required |
| G-03 | DR-003 unit-tests: PASS | Required |
| G-04 | DR-004 race-tests: PASS | Required |
| G-05 | DR-005 e2e-tests: PASS or SKIP (with documented reason) | SKIP acceptable |
| G-06 | DR-006 prove-gate: PASS (overall=PASS, 0 FAIL) | Required |
| G-07 | DR-007 verify-release: PASS (0 FAIL; V-007..V-009 may SKIP) | Required (FAILs not tolerated) |
| G-08 | DR-008 install-drill: PASS | Required |
| G-09 | DR-009 rollback-drill: PASS or SKIP (with documented reason) | SKIP acceptable |
| G-10 | CI `patch-dry-run.yml` workflow: green | Required |
| G-11 | No blocking comments on patch PR | Required |
| G-12 | Release record `docs/runbooks/releases/v1.2.1.yaml` created | Required before tag |

**Decision:** If G-01..G-04, G-06..G-08, G-10, G-11, G-12 are all PASS and G-05/G-09 are PASS
or documented SKIP (with reason), the release is eligible for promotion. G-12 must be PASS —
the release record must exist before the tag is pushed.

## Live Promotion Steps (after dry-run passes)

These steps are executed only after all go/no-go gates above are green.

```bash
# 1. Ensure on main, up to date
git checkout main && git pull

# 2. Set version
VERSION=v1.2.1
RELEASE_DIR=/srv/storage/shared/tools/releases/codero-${VERSION}
mkdir -p "${RELEASE_DIR}"

# 3. Build with real version tag
go build -trimpath \
  -ldflags "-s -w -X main.version=${VERSION}" \
  -o "${RELEASE_DIR}/codero" \
  ./cmd/codero

# 4. Record checksum
sha256sum "${RELEASE_DIR}/codero" | tee "${RELEASE_DIR}/codero.sha256"

# 5. Back up current binary (rollback anchor)
cp -p /srv/storage/shared/tools/bin/codero \
      /srv/storage/shared/tools/bin/codero.prev.bak

# 6. Atomic install
cp "${RELEASE_DIR}/codero" /srv/storage/shared/tools/bin/codero.new
mv /srv/storage/shared/tools/bin/codero.new /srv/storage/shared/tools/bin/codero

# 7. Verify
codero version   # must print v1.2.1

# 8. Tag
git tag v1.2.1 -m "v1.2.1 — <one-line summary>"
git push origin v1.2.1

# 9. Post-install verification
./scripts/automation/verify-release.sh \
  --version v1.2.1 \
  --repo-path . \
  --fast
```

## Rollback Procedure

```bash
# Atomic rollback: stage the backup in the same directory, then rename into place
# to avoid a partially-written live binary.
BINDIR=/srv/storage/shared/tools/bin

cp -p "${BINDIR}/codero.prev.bak" "${BINDIR}/codero.rollback.tmp"
chmod --reference="${BINDIR}/codero" "${BINDIR}/codero.rollback.tmp"
mv "${BINDIR}/codero.rollback.tmp" "${BINDIR}/codero"

codero version   # confirm previous version

# If the daemon was running, restart it after rollback:
# codero stop && codero start
```

Rollback is always safe because:
- The `mv` within the same filesystem is atomic (kernel-level rename).
- No database migrations run during a patch install.
- `codero.prev.bak` is written before the new binary goes live.
